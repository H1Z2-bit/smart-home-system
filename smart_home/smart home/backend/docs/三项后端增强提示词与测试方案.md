本文档用于推进当前智能家居 FastAPI 后端的三个小增强：

1. 设备控制写入 `device_event`
2. 设备模拟字段完整入库
3. 操作日志使用当前登录用户

当前项目状态：

- MySQL 模式已联调通过。
- `python scripts\mysql_smoke_test.py` 已通过。
- `python -m pytest tests\test_mysql_mode_flow.py -q` 已通过。
- 房间、设备、控制、模拟、报警、自检、日志等基础接口已经可用。

当前实施状态：

- 增强 1 已完成：设备控制成功后会写入 `device_event`。
- 增强 2 已完成：设备模拟字段已完整写入并返回。
- 增强 3 已完成：用户主动操作的日志、设备事件、报警处理、自检记录已尽量使用当前登录用户 ID。
- 最新验证结果：MySQL 联调用例 `5 passed`，全量 pytest `30 passed, 5 skipped`。

建议按本文档顺序逐项完成，每完成一项就跑一次测试，避免多个改动叠在一起不好排查。

## 一、增强 1：设备控制写入 device_event

### 1.1 目标

当前设备控制接口：

```text
POST /api/devices/{device_id}/control
```

已经能修改设备状态，并写入 `operation_log`。

本次增强目标是在设备控制成功后，同时写入 `device_event` 表，记录设备状态变化。

### 1.2 实现提示词

```text
请在当前 FastAPI 智能家居后端中实现“设备事件记录”增强。

当前项目状态：
1. 后端使用 FastAPI + SQLAlchemy + MySQL。
2. 已完成 MySQLRepository 模式联调。
3. 已完成设备控制接口 POST /api/devices/{device_id}/control。
4. 数据库中已经存在 device_event 表。
5. 当前设备控制成功后会写 operation_log，但还没有写 device_event。

本次目标：
设备控制成功后，在 device_event 中新增一条设备事件记录。

要求：
1. 新增或完善 DeviceEventRepository 抽象。
   建议文件：
   - backend/app/repositories/device_event_repository.py

2. 新增 MockDeviceEventRepository。
   建议文件：
   - backend/app/repositories/mock/device_event_mock.py

3. 新增 MySQLDeviceEventRepository。
   建议文件：
   - backend/app/repositories/mysql/device_event_mysql.py

4. 修改 repositories/factory.py，增加：
   - get_device_event_repository()

5. device_event 写入字段：
   - home_id
   - device_id
   - event_type
   - old_status
   - new_status
   - event_desc
   - operator_id
   - created_at 由数据库或 Repository 自动生成

6. 设备控制成功后写入事件：
   - event_type 使用 control
   - old_status 为控制前设备状态
   - new_status 为控制后的设备状态
   - event_desc 示例：控制设备 客厅灯：off -> on
   - operator_id 暂时可先沿用当前逻辑中的默认用户 1；如果已经实现当前用户透传，则使用当前登录用户 ID

7. 不要把 SQL 写在 Controller 层。
8. 保持 Router -> Service -> Repository 分层。
9. Mock 模式和 MySQL 模式都要可用。
10. 补充 pytest 测试：
    - 控制设备后，device_event 记录数量增加。
    - 最新事件 event_type 为 control。
    - old_status 和 new_status 正确。
11. 更新或补充接口测试文档，说明如何验证 device_event。
12. 保持现有测试全部通过。
```

## 二、增强 2：设备模拟字段完整入库

### 2.1 目标

当前设备模拟接口：

```text
POST /api/devices/{device_id}/simulate
```

已经能写入模拟记录，并可触发报警。

但需要进一步确认并补齐以下字段完整入库：

- `trigger_alarm`
- `alarm_type`
- `alarm_level`
- `simulation_type`
- `scenario_name`

### 2.2 实现提示词

```text
请在当前 FastAPI 智能家居后端中完善“设备模拟字段完整入库”。

当前项目状态：
1. 已完成设备模拟接口 POST /api/devices/{device_id}/simulate。
2. 已完成查询设备模拟记录 GET /api/devices/{device_id}/simulations。
3. 数据库 device_simulation 表已经包含：
   - simulation_id
   - home_id
   - device_id
   - metric_name
   - metric_value
   - device_status
   - trigger_alarm
   - alarm_type
   - alarm_level
   - alarm_id
   - simulation_type
   - scenario_name
   - created_at
4. 当前后端已能根据 trigger_alarm 创建 alarm_record。

本次目标：
确保设备模拟接口请求体中的模拟相关字段完整保存到 device_simulation 表，并在查询模拟记录时完整返回。

要求：
1. 检查并完善 backend/app/schemas/device_simulation.py。
   DeviceSimulationCreate 建议字段：
   - metric_name
   - metric_value
   - device_status
   - trigger_alarm
   - alarm_type
   - alarm_level
   - simulation_type
   - scenario_name

2. simulation_type 默认值为 manual。
   可选值建议：
   - manual
   - auto
   - scenario

3. scenario_name 可为空。

4. 修改 device_simulation_service.py：
   创建模拟记录时，需要把以下字段传入 Repository：
   - trigger_alarm
   - alarm_type
   - alarm_level
   - simulation_type
   - scenario_name
   - alarm_id

5. MockDeviceSimulationRepository 和 MySQLDeviceSimulationRepository 都要支持这些字段。

   - 创建 alarm_record。
   - device_simulation.alarm_id 保存对应报警 ID。
   - device_simulation.trigger_alarm 保存为 true。
   - alarm_type、alarm_level 与请求体保持一致。

7. 如果 trigger_alarm 为 false：
   - 不创建 alarm_record。
   - alarm_id 为 null。
   - trigger_alarm 保存为 false。

8. 保持统一响应格式。
9. 补充 pytest 测试：
   - 触发报警时，模拟记录 trigger_alarm 为 true。
   - alarm_type、alarm_level 正确入库并返回。
   - simulation_type、scenario_name 正确入库并返回。
   - 不触发报警时 alarm_id 为 null。
10. 保持现有测试全部通过。
```

## 三、增强 3：操作日志使用当前登录用户

### 3.1 目标

当前部分业务日志仍可能使用：

```text
DEFAULT_OPERATOR_ID = 1
```

这适合早期 Mock 阶段，但 MySQL 联调和多用户场景下不够准确。

本次增强目标是：房间、设备、控制、模拟、情景、定时、联动、报警、自检等操作日志中的 `operator_id` 使用当前登录用户 ID。

### 3.2 实现提示词

```text
请在当前 FastAPI 智能家居后端中实现“操作日志使用当前登录用户”增强。

当前项目状态：
1. 已完成登录认证。
2. API 路由中已经可以通过 Depends(get_current_user) 获取 current_user。
3. operation_log 表中有 operator_id 字段。
4. 目前部分 Service 中仍使用 DEFAULT_OPERATOR_ID = 1 写日志。

本次目标：
让业务操作日志、设备事件、报警处理、自检记录等操作人字段尽量使用当前登录用户 ID，而不是固定写 1。

要求：
1. 检查以下 Service：
   - room_service.py
   - device_service.py
   - device_simulation_service.py
   - scene_service.py
   - schedule_service.py
   - linkage_service.py
   - alarm_service.py
   - self_check_service.py

2. 对会写 operation_log 的方法增加 operator_id 参数。
   示例：
   - create_room(home_id, data, operator_id)
   - update_room(room_id, data, operator_id)
   - delete_room(room_id, operator_id)
   - control_device(device_id, data, operator_id)
   - create_simulation(device_id, data, operator_id)

3. 修改对应 Router：
   - 通过 current_user: TokenPayload = Depends(get_current_user) 获取当前用户。
   - 调用 Service 时传入 current_user.user_id。

4. 移除或减少 DEFAULT_OPERATOR_ID 的使用。
   如果某些系统自动行为没有登录用户，可以保留 operator_id=None 或 system operator，但普通用户主动操作必须使用 current_user.user_id。

5. 报警处理字段：
   - confirmed_by
   - processed_by
   应尽量使用 current_user.user_id。

6. device_event.operator_id 也应使用当前登录用户。

7. 保持接口路径和请求体不变。
8. 保持统一响应格式。
9. 补充 pytest 测试：
   - 使用业主账号登录并控制设备，最新 operation_log.operator_id 等于业主 user_id。
   - 使用另一个账号登录执行可授权操作时，operator_id 等于该账号 user_id。
   - 报警处理后 processed_by 等于当前用户 ID。
10. 保持现有测试全部通过。
```

## 四、推荐实施顺序

### 第一步：设备模拟字段完整入库

优先级最高，原因：

- 改动范围相对小。
- 数据库字段已经存在。
- 很容易通过 MySQL 查询验证。
- 和报警演示直接相关。

### 第二步：设备控制写入 device_event

原因：

- 展示效果明显。
- 能支撑设备详情页“设备动态”。
- 后续联动、定时、报警控制设备时也能复用。

### 第三步：操作日志使用当前登录用户

原因：

- 改动涉及 Router 和多个 Service，范围稍大。
- 适合在前两项稳定后再统一整理。
- 做完后系统的审计链路会更完整。
