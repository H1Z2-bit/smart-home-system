# 智能家居后端接口测试详细步骤

本文档用于在 MySQL 模式下手动测试当前 FastAPI 后端接口，适合课程设计联调、答辩演示前自检，以及前后端对接前确认接口状态。

## 1. 测试前准备

### 1.1 确认后端目录

```powershell
cd "D:\coding\VScode\projects\smart_home\smart home\backend"
```

### 1.2 激活虚拟环境

```powershell
.\.venv\Scripts\Activate.ps1
```

如果 PowerShell 提示脚本执行被禁止，先执行：

```powershell
Set-ExecutionPolicy -Scope Process -ExecutionPolicy Bypass
```

然后重新激活虚拟环境。

### 1.3 检查 `.env`

确认 `backend/.env` 中至少包含：

```env
DATABASE_HOST=127.0.0.1
DATABASE_PORT=3306
DATABASE_NAME=smart_home
DATABASE_USER=root
DATABASE_PASSWORD=你的MySQL密码
USE_MOCK_REPOSITORY=false
```

`USE_MOCK_REPOSITORY=false` 表示接口使用 MySQL 数据库，而不是 Mock 内存数据。

### 1.4 导入数据库

如需重新初始化数据库，执行：

```powershell
.\scripts\import_mysql_schema.ps1 `
  -MysqlExe "D:\coding\MySQL\bin\mysql.exe" `
  -SqlFile "D:\coding\VScode\projects\smart_home\smart_home.sql" `
  -User root
```

成功时应看到：

```text
MySQL schema imported successfully.
```

### 1.5 先跑自动化联调

冒烟测试：

```powershell
python scripts\mysql_smoke_test.py
```

成功结果：

```text
MySQL smoke test passed.
```

完整 MySQL 联调：

```powershell
$env:USE_MOCK_REPOSITORY="false"
$env:RUN_MYSQL_TESTS="1"
python -m pytest tests\test_mysql_mode_flow.py -q
```

成功结果：

```text
5 passed
```

python -m uvicorn app.main:app --host 0.0.0.0 --port 8000 --reload

启动后访问：

健康检查：http://127.0.0.1:8000/api/health

## 3. Swagger 测试步骤

### 3.1 打开接口文档

浏览器访问：

```text
http://127.0.0.1:8000/docs
```

### 3.2 登录获取 token

找到：

```text
POST /api/auth/login
```

点击 `Try it out`，请求体填写：

```json
{
  "phone": "13800000000",
  "password": "123456"
}
```

成功返回中会有：

```json
{
  "code": 200,
  "message": "success",
  "data": {
    "token": "..."
  }
}
```

复制 `data.token`。

### 3.3 配置 Swagger 鉴权

点击 Swagger 页面右上角 `Authorize`，填写：

```text
Bearer 复制到的token
```

注意 `Bearer` 和 token 中间有一个空格。

之后再测试房间、设备、报警、日志等接口时，Swagger 会自动携带认证头。

## 4. 推荐测试顺序

建议按下面顺序测试，这样出现问题时更容易定位。

### 4.1 健康检查

接口：

```text
GET /api/health
```

预期结果：

```json
{
  "code": 200,
  "message": "success",
  "data": {
    "status": "ok"
  }
}
```

### 4.2 用户与家庭空间

接口：

```text
GET /api/users/profile
GET /api/homes
GET /api/homes/1/system/config
```

预期：

- 当前用户手机号为 `13800000000`。
- 家庭空间中存在 `演示家庭`。
- 系统配置能正常返回。

### 4.3 房间管理

查询房间：

```text
GET /api/homes/1/rooms
```

新增房间：

```text
POST /api/homes/1/rooms
```

请求体：

```json
{
  "room_name": "测试书房",
  "room_type": "study",
  "remark": "接口测试房间"
}
```

修改房间：

```text
PUT /api/rooms/{room_id}
```

请求体：

```json
{
  "room_name": "测试书房A",
  "room_type": "study",
  "remark": "已修改"
}
```

删除房间：

```text
DELETE /api/rooms/{room_id}
```

注意：如果房间下仍有设备，应返回：

```json
{
  "code": 400,
  "message": "该房间下存在设备，不能删除",
  "data": null
}
```

### 4.4 设备管理

查询设备：

```text
GET /api/homes/1/devices
```

按房间筛选：

```text
GET /api/homes/1/devices?room_id=1
```

查询设备详情：

```text
GET /api/devices/1
```

新增设备：

```text
POST /api/homes/1/devices
```

请求体：

```json
{
  "room_id": 1,
  "device_name": "测试台灯",
  "device_type": "light",
  "device_status": "off",
  "is_key_device": false
}
```

修改设备：

```text
PUT /api/devices/{device_id}
```

请求体：

```json
{
  "device_name": "测试台灯A",
  "device_type": "light",
  "device_status": "off",
  "is_key_device": false
}
```

删除设备：

```text
DELETE /api/devices/{device_id}
```

### 4.5 设备控制

控制灯或插座：

```text
POST /api/devices/{device_id}/control
```

请求体：

```json
{
  "action": "switch",
  "target_state": "on",
  "param_value": null
}
```

预期：

- `light`、`socket` 类型设备可控制。
- 控制成功后返回更新后的设备信息。
- `device_status` 应变为 `on` 或 `off`。
- 操作日志中应新增 `device_control` 记录。
- MySQL 的 `device_event` 表中应新增一条 `control` 事件。
- `operation_log.operator_id` 和 `device_event.operator_id` 应等于当前登录用户 ID。

异常测试：

```text
POST /api/devices/4/control
```

如果设备 4 是烟雾传感器，应返回：

```json
{
  "code": 400,
  "message": "传感器类设备不允许控制",
  "data": null
}
```

### 4.6 设备模拟

模拟烟雾数据并触发报警：

```text
POST /api/devices/4/simulate
```

请求体：

```json
{
  "metric_name": "smoke",
  "metric_value": 92,
  "device_status": "online",
  "trigger_alarm": true,
  "alarm_type": "smoke",
  "alarm_level": "serious",
  "simulation_type": "scenario",
  "scenario_name": "厨房烟雾升高演示"
}
```

预期：

- 返回模拟记录。
- `alarm_id` 不为空。
- `trigger_alarm` 为 `true`。
- `alarm_type`、`alarm_level`、`simulation_type`、`scenario_name` 与请求体一致。
- 报警列表中新增一条报警记录。
- 操作日志中新增 `device_simulate` 记录。

查询模拟记录：

```text
GET /api/devices/4/simulations
```

### 4.7 报警处理

查询报警列表：

```text
GET /api/homes/1/alarms
```

查询报警详情：

```text
GET /api/alarms/{alarm_id}
```

确认报警：

```text
POST /api/alarms/{alarm_id}/confirm
```

请求体：

```json
{
  "process_desc": "已收到报警，正在确认现场情况",
  "process_result": "success"
}
```

处理报警：

```text
POST /api/alarms/{alarm_id}/process
```

请求体：

```json
{
  "process_desc": "已检查厨房烟雾情况",
  "process_result": "success"
}
```

预期：

- 报警状态变为 `processing`。
- `processed_by` 应等于当前登录用户 ID。
- 报警处理日志中的处理人也应等于当前登录用户 ID。

关闭报警：

```text
POST /api/alarms/{alarm_id}/resolve
```

请求体：

```json
{
  "process_desc": "现场已处理完成",
  "process_result": "success"
}
```

标记误报：

```text
POST /api/alarms/{alarm_id}/false-alarm
```

请求体：

```json
{
  "process_desc": "确认是误报",
  "process_result": "false_alarm"
}
```

### 4.8 设备自检

发起自检：

```text
POST /api/devices/4/self-check
```

预期：

- 返回一条自检记录。
- `check_result` 正常情况下为 `normal`。
- `operator_id` 应等于当前登录用户 ID。
- 操作日志中新增 `device_self_check` 记录。

查询自检记录：

```text
GET /api/devices/4/self-checks
```

### 4.9 情景模式

### 4.12 操作日志

查询业务操作日志：

```text
GET /api/homes/1/logs
```

重点检查是否包含：

- `room_create`
- `room_update`
- `room_delete`
- `device_create`
- `device_update`
- `device_delete`
- `device_control`
- `device_simulate`
- `device_self_check`

同时检查用户主动操作产生的日志：

- `operator_id` 应等于当前登录用户 ID。
- 使用 `13800000000 / 123456` 登录时，演示数据中通常对应 `operator_id = 1`。
- 使用其他有权限账号操作时，`operator_id` 应随登录用户变化。

查询系统/账号日志：

```text
GET /api/homes/1/system/logs
```

系统日志主要用于登录、成员、系统配置等相关操作。

## 6. 数据库验证 SQL

如果想确认接口是否真正写入 MySQL，可以进入 MySQL：

```powershell
& "D:\coding\MySQL\bin\mysql.exe" -uroot -p
```

然后执行：

```sql
USE smart_home;
```

查看设备状态：

```sql
SELECT device_id, device_name, device_type, device_status
FROM device
ORDER BY device_id;
```

查看操作日志：

```sql
SELECT log_id, operator_id, operation_type, operation_object, operation_result, operation_desc, created_at
FROM operation_log
ORDER BY log_id DESC
LIMIT 10;
```

查看设备事件：

```sql
SELECT event_id, home_id, device_id, event_type, old_status, new_status, event_desc, operator_id, created_at
FROM device_event
ORDER BY event_id DESC
LIMIT 10;
```

查看设备模拟记录：

```sql
SELECT simulation_id, home_id, device_id, metric_name, metric_value,
       device_status, trigger_alarm, alarm_type, alarm_level,
       alarm_id, simulation_type, scenario_name, created_at
FROM device_simulation
ORDER BY simulation_id DESC
LIMIT 10;
```

查看报警记录：

```sql
SELECT alarm_id, home_id, device_id, alarm_type, alarm_level, alarm_status,
       confirmed_by, processed_by, trigger_value, trigger_time, confirmed_at, processed_at
FROM alarm_record
ORDER BY alarm_id DESC
LIMIT 10;
```

查看报警处理日志：

```sql
SELECT process_id, alarm_id, processor_id, process_desc, process_result, processed_at
FROM alarm_process_log
ORDER BY process_id DESC
LIMIT 10;
```

## 7. 常见问题排查

### 7.1 返回 401 unauthorized

原因通常是没有携带 token，或 token 格式错误。

检查请求头：

```text
Authorization: Bearer token内容
```

### 7.2 返回 403 forbidden

原因通常是当前账号没有对应家庭空间或模块权限。

建议先使用演示业主账号：

```text
13800000000 / 123456
```

### 7.3 返回数据库连接错误

优先检查：

- MySQL 服务是否启动。
- `.env` 中 `DATABASE_PASSWORD` 是否正确。
- `.env` 中 `USE_MOCK_REPOSITORY=false` 是否正确。
- `smart_home.sql` 是否已经成功导入。

### 7.4 Swagger Authorize 后仍然 401

确认填写的是：

```text
Bearer eyJ...
```

不是只填 token，也不是填：

```text
Authorization: Bearer eyJ...
```

### 7.5 测试数据被改乱

重新导入数据库：

```powershell
.\scripts\import_mysql_schema.ps1 `
  -MysqlExe "D:\coding\MySQL\bin\mysql.exe" `
  -SqlFile "D:\coding\VScode\projects\smart_home\smart_home.sql" `
  -User root
```

然后重新执行自动化联调：

```powershell
python scripts\mysql_smoke_test.py
$env:USE_MOCK_REPOSITORY="false"
$env:RUN_MYSQL_TESTS="1"
python -m pytest tests\test_mysql_mode_flow.py -q
```

## 9. 当前阶段测试结论标准

如果满足以下条件，可以认为当前后端接口联调基本通过：

- `python scripts\mysql_smoke_test.py` 通过。
- `python -m pytest tests\test_mysql_mode_flow.py -q` 通过。
- Swagger 中登录、房间、设备、设备控制、设备模拟、报警、自检、日志接口可手动调用。
- MySQL 中能看到对应的设备状态变化、设备事件、模拟记录、报警记录和操作日志。
- 用户主动操作产生的 `operation_log.operator_id`、`device_event.operator_id`、报警处理人字段、自检 `operator_id` 与当前登录用户一致。

完成以上测试后，可以进入下一阶段：定时任务执行器、联动规则触发器、设备事件查询接口、前后端接口联调与答辩演示数据整理。
